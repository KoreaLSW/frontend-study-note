# CSR의 SEO 취약점과 최신 해결 기술 (SSR, SSG, Hydration)

## 1. CSR이 SEO(검색 엔진 최적화)에 취약한 이유 3가지

검색 엔진 로봇(크롤러)의 관점에서 CSR은 **"보물찾기를 하는데 지도가 백지인 상황"**과 같습니다.

### ① "빈 껍데기" 문제 (Empty Content)

- **서버가 주는 것**: `<div id="root"></div>`와 같이 내용이 없는 빈 HTML.
- **문제점**: 검색 봇은 HTML을 받자마자 분석하는데, 내용이 없으므로 **"정보가 없는 페이지"**로 판단하고 떠나버림.

### ② "기다려주지 않음" (Timing Issue)

- **문제점**: 구글 봇 등은 JS 실행 능력이 있지만, 데이터를 받아오고 화면을 그리는 **수 초의 시간을 기다려주지 않음**.
- 스크린샷을 찍듯 초기 상태만 훑고 지나갈 확률이 높음.

### ③ SNS 공유 미리보기 실패 (Open Graph)

- **문제점**: 카카오톡, 슬랙 등의 봇은 **자바스크립트를 실행하지 못함**.
- HTML 헤더(`<head>`)의 정보만 읽어가는데, CSR은 이 정보도 나중에 채워지므로 썸네일이나 제목이 뜨지 않음.

---

## 2. 최신 해결책: "하이브리드 렌더링" (Next.js 등)

SPA의 장점(UX)과 검색 노출(SEO)을 모두 잡기 위한 기술의 진화입니다.

### ① SSR (Server Side Rendering)

- **방식**: "첫 페이지는 서버가 완성해서 주자!"
- 접속 시 서버에서 **데이터가 채워진 완성된 HTML**을 전송.
- **결과**: 검색 봇이 내용을 바로 읽을 수 있어 **SEO 만점**. 이후 페이지 이동은 CSR처럼 동작(Universal Rendering).

### ② SSG (Static Site Generation)

- **방식**: "내용이 안 바뀌면 미리 만들어두자."
- 빌드 타임에 **HTML 파일 수백 개를 미리 생성(Generate)**.
- **결과**: 가장 빠르고 SEO도 완벽함 (블로그, 소개 페이지 등에 적합).

### ③ 핵심 기술: Hydration

1. **Dry HTML**: 서버에서 데이터가 채워진 **정적인 HTML**을 먼저 보냄 (SEO 해결, 화면 바로 뜸).
2. **Hydration**: 브라우저가 그 위에 **JS(React 등)**를 실행하여 이벤트 리스너 등을 연결.
3. **Interactive App**: 정적이었던 화면이 **동적인 SPA**로 변신하여 작동.

## 3. 요약 비교표

| 구분          | 과거 (Pure CSR)       | 현재 (Modern Hybrid)       |
| :------------ | :-------------------- | :------------------------- |
| **HTML 상태** | 텅 빈 껍데기          | **데이터가 채워진 완성본** |
| **검색 로봇** | "내용 없음" (무시)    | **"내용 풍부함" (수집)**   |
| **SNS 공유**  | 썸네일/제목 누락      | **썸네일/제목 정상 표시**  |
| **기술 핵심** | Client Side Rendering | **SSR / SSG + Hydration**  |
