# SPA(Single Page Application)와 CSR(Client Side Rendering) 핵심 정리

## 1. SPA (Single Page Application)

- **정의**: 말 그대로 페이지가 하나뿐인 웹 애플리케이션.
- **특징**: 초반에 한 번만 페이지를 받아오고, 그 후 새로운 페이지를 전체적으로 받아오지 않음.
- **결과**: 버튼을 누르거나 페이지가 전환될 때 화면 전체를 새로고침(깜빡임)하지 않고 **즉각적으로 반응**을 함.

---

## 2. CSR (Client Side Rendering)

- **정의**: 최초 접속 때 웹사이트 전체 리소스(HTML, CSS, JS)를 한 번에 받아와 브라우저에서 그리는 방식.

### 🔄 동작 (로딩)

1. **초기 로딩**: 최초 접속 때 웹사이트 전체 리소스(HTML, CSS, JS)를 한 번에 받아옴.
2. **화면 갱신**: 이후 필요한 데이터(JSON)만 서버에서 받아와 JS(브라우저)가 화면의 일부분을 갈아 끼움.

### ✅ 장점

- **서버 부하 감소**: 서버에서 HTML 전체를 만들어 보내줄 필요 없이 필요한 데이터만 보내기 때문에 서버의 부담을 줄일 수 있음.
- **협업 용이**: 프론트엔드와 백엔드 분리로 개발 업무 분업화 및 협업이 용이함.

### ⚠️ 단점

- **초기 로딩 속도가 느림**: 최초 로딩 시 필요한 리소스를 한 번에 받아와야 하기 때문.
- **SEO(검색 엔진 최적화)의 어려움**: 검색 엔진 로봇이 접속했을 때, 데이터가 없는 **빈 HTML**만 보일 수 있음.

---

## 3. CSR 렌더링 순서

> **핵심**: 서버는 **빈 껍데기**만 주고, 내용물은 **브라우저(JS)**가 나중에 채운다.

1. **서버 요청 (Request)**: 사용자가 사이트 접속.
2. **빈 HTML 수신**: 서버로부터 비어있는 뼈대 HTML (`<div id="root">`) 수신.
3. **JS 다운로드**: HTML에 연결된 JS 파일 다운로드.
4. **JS 실행**: 브라우저가 JS 실행.
5. **데이터 요청**: JS가 서버 API에 데이터(JSON) 요청.
6. **렌더링 (Rendering)**: 받아온 데이터를 HTML에 결합하여 화면 표시.

### ⏳ 요약표: 누가 언제 일하나?

| 순서 | 주체         | 하는 일                | 사용자 화면 상태      |
| :--- | :----------- | :--------------------- | :-------------------- |
| 1    | **브라우저** | 서버 접속              | 로딩 중               |
| 2    | **서버**     | 빈 HTML 전달           | ⚪️ 하얀 화면 (Blank) |
| 3    | **브라우저** | JS 파일 다운로드       | ⚪️ 하얀 화면 (Blank) |
| 4    | **브라우저** | JS 실행 및 데이터 요청 | ⚪️ 하얀 화면 (Blank) |
| 5    | **서버**     | 데이터(JSON) 전달      | ⚪️ 하얀 화면 (Blank) |
| 6    | **브라우저** | 화면 조립 및 표시      | ✨ 완성된 화면        |
