# SSG (Static Site Generation)란?

SSG는 **빌드 타임에 서버에서 미리 렌더링**하는 방식입니다.

## 언제 렌더링되나요?

SSG는 처음 한 번 빌드할 때 모든 페이지를 미리 렌더링합니다. 정적인 사이트를 만들 때 주로 사용하는 방식입니다.

사용자가 다음에 또 접근할 때는 서버까지 가지 않고, 이미 생성된 정적 파일을 CDN에서 빠르게 가져올 수 있습니다.

## 장점

- **페이지 로딩 시간이 빠름**: 이미 렌더링된 HTML을 제공하므로 TTV가 매우 빠릅니다
- **SEO 최적화가 좋음**: 검색 엔진이 완전히 렌더링된 HTML을 크롤링할 수 있습니다
- **보안이 뛰어남**: 서버 사이드 로직이 없어 공격 표면이 줄어듭니다
- **CDN 캐시 가능**: 정적 파일이므로 CDN에 캐싱하여 전 세계 어디서나 빠르게 제공할 수 있습니다

## 단점

- **정적인 데이터만 출력됨**: 빌드 시점의 데이터만 포함되므로 실시간 데이터 반영이 어렵습니다
- **사용자별 정보 제공이 어려움**: 모든 사용자에게 동일한 페이지를 제공하므로 개인화된 콘텐츠 제공에 한계가 있습니다

## 사용 사례

문서 사이트 및 제품 소개 페이지에 특히 용이합니다. 콘텐츠가 자주 변경되지 않고, 모든 사용자에게 동일한 정보를 제공하는 경우에 적합합니다.

# ISR (Incremental Static Regeneration)란?

ISR은 렌더링 방식이 기본적으로 SSG와 동일하나, **주기적으로 재렌더링이 가능**한 방식입니다.

## 언제 렌더링되나요?

ISR은 처음 빌드할 때 페이지를 생성하고, 이후 설정한 주기(예: 60초)마다 백그라운드에서 페이지를 재생성합니다. 사용자가 요청할 때 기존 페이지를 먼저 보여주고, 백그라운드에서 업데이트된 페이지를 생성하여 다음 요청부터는 새 페이지를 제공합니다.

## 장점

- **SSG의 모든 장점**: 페이지 로딩 시간이 빠르고, SEO 최적화가 좋으며, CDN 캐시가 가능합니다
- **데이터가 주기적으로 업데이트됨**: 빌드 후에도 일정 주기마다 데이터를 갱신할 수 있어, SSG의 단점인 정적 데이터 문제를 부분적으로 해결합니다

## 단점

- **실시간 데이터가 아님**: 주기적으로 데이터를 불러올 수 있지만, 설정한 주기 동안은 이전 데이터가 표시되므로 완전한 실시간 반영은 불가능합니다
- **사용자별 정보 제공이 어려움**: SSG와 마찬가지로 모든 사용자에게 동일한 페이지를 제공하므로 개인화된 콘텐츠 제공에는 한계가 있습니다

## 사용 사례

블로그, 제품 카탈로그 등 콘텐츠가 주기적으로 업데이트되지만 실시간일 필요는 없는 경우에 적합합니다.

# SSR (Server Side Rendering)란?

SSR은 **렌더링하는 주체가 서버**이고, **클라이언트에서 요청 시마다 렌더링**하는 방식입니다.

## 언제 렌더링되나요?

SSR은 사용자가 페이지를 요청할 때마다 서버에서 데이터를 가져와 HTML을 생성하여 클라이언트에 전달합니다. 매 요청마다 서버에서 실시간으로 렌더링이 이루어집니다.

## 장점

- **페이지 로딩 시간(TTV)이 빠름**: 서버에서 완전히 렌더링된 HTML을 제공하므로 초기 로딩이 빠릅니다
- **SEO 최적화가 좋음**: 검색 엔진이 완전히 렌더링된 HTML을 크롤링할 수 있습니다
- **보안이 뛰어남**: 서버 사이드에서 데이터를 처리하므로 민감한 로직을 클라이언트에 노출하지 않습니다
- **실시간 데이터를 사용**: 매 요청마다 최신 데이터를 가져와서 렌더링하므로 실시간 데이터 반영이 가능합니다
- **사용자별 필요한 데이터를 사용 가능**: 요청 시 사용자 정보를 기반으로 개인화된 콘텐츠를 제공할 수 있습니다

## 단점

- **서버의 과부하가 걸릴 수 있음**: 매 요청마다 서버에서 렌더링을 수행하므로 트래픽이 많을 경우 서버 부하가 증가합니다
- **CDN에 캐시가 어려움**: 매 요청마다 다른 내용을 생성할 수 있어 CDN 캐싱이 제한적입니다

## 사용 사례

실시간 데이터가 중요하거나 사용자별로 다른 콘텐츠를 제공해야 하는 경우(예: 대시보드, 개인화된 쇼핑몰, 소셜 미디어 피드)에 적합합니다.

CSR (Client Side Rendering)란?

CSR은 **렌더링하는 주체가 클라이언트(브라우저)**이고, **브라우저에서 JavaScript가 실행될 때 렌더링**하는 방식입니다.

## 언제 렌더링되나요?

CSR은 사용자가 페이지를 요청하면 서버는 빈 HTML 뼈대만 전달하고, 브라우저에서 JavaScript를 다운로드한 후 실행하여 화면을 렌더링합니다. 이후 페이지 전환 시에도 서버에 요청하지 않고 클라이언트에서만 처리합니다.

## 장점

- **서버 부하 감소**: 서버에서 HTML을 생성할 필요 없이 필요한 데이터(JSON)만 전달하면 되므로 서버 부담이 적습니다
- **협업 용이**: 프론트엔드와 백엔드를 완전히 분리하여 개발할 수 있어 협업이 용이합니다
- **페이지 전환 시 깜빡임 없음**: 페이지 전환 시 전체 새로고침 없이 필요한 부분만 업데이트하여 부드러운 사용자 경험을 제공합니다
- **인터랙티브한 사용자 경험**: 클라이언트에서 즉각적인 반응을 제공할 수 있습니다

## 단점

- **초기 로딩 속도가 느림**: 최초 로딩 시 모든 JavaScript 파일을 다운로드하고 실행해야 하므로 초기 로딩 시간이 길어집니다
- **SEO(검색 엔진 최적화)의 어려움**: 검색 엔진 로봇이 접속했을 때 빈 HTML만 보이므로 콘텐츠를 크롤링하기 어렵습니다
- **빈 HTML 문제**: 서버가 `<div id="root"></div>`와 같은 빈 껍데기만 전달하므로, JavaScript가 실행되기 전까지는 화면에 아무 내용도 표시되지 않습니다

## 사용 사례

SEO가 중요하지 않고, 사용자 인터랙션이 많은 애플리케이션에 적합합니다.
